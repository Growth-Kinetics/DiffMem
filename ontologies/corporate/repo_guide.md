# DiffMem Corporate Repository Guide

This guide defines the structure, creation, and maintenance of a DiffMem memory
repository using the **corporate ontology**. Memory files are plain-text Markdown
in a Git repo. Git's commit history captures how knowledge evolves over time.

## Architecture

Same as the base DiffMem architecture: one orphan branch per agent/user, checked
out into an isolated worktree. The difference is the folder taxonomy and entity
semantics.

## Repository Structure

```
<worktree_root>/
├── repo_guide.md          # This file (schema reference)
├── index.md               # Autogenerated: entity manifest with SEMANTIC INDEX
├── {user_id}.md           # Agent/team root entity (always present)
├── entities/
│   ├── people/            # Individual persons (internal and external contacts)
│   ├── projects/          # Active or historical projects
│   ├── decisions/         # Architectural, strategic, or operational decisions (ADR-style)
│   ├── commitments/       # Promises, deliverables, obligations, milestones
│   └── external/          # External organizations, vendors, clients, APIs
└── timeline/
    └── YYYY-MM.md         # Monthly chronological session log
```

## File Naming Convention

Files use **snake_case** (all lowercase, words separated by underscores). The engine applies this automatically:
- `entities/people/alice_smith.md`
- `entities/projects/diff_mem.md`
- `entities/decisions/decide_git_storage.md`
- `entities/commitments/deliver_v1_api.md`
- `entities/external/stripe_payments.md`

## Linking Between Entities

Interactions with other entities MUST be declared using **Obsidian-style wikilinks**:
```
[[entity_name]]
```
Examples inside `entities/projects/diff_mem.md`:
- `[[alice_smith]]` (a person)
- `[[decide_git_storage]]` (a decision)
- `[[deliver_v1_api]]` (a commitment)

One wikilink per target entity per file. Link the first or most prominent mention.
The `## SEMANTIC INDEX` section at the bottom of each file is off-limits to link insertion.

## Entity Types and Templates

### 1. People (`entities/people/`)

Individuals — internal team members, external contacts, stakeholders.

```markdown
# Person: [Name]

## Core Metadata
- Role: [e.g., Lead Engineer, Co-Founder]
- Contact: [e.g., email@domain.com]
- Affiliation: [Internal or External]

## Active Context & Relationships
- Projects involved: [[project_name]]
- Key commitments: [[commitment_name]]

## Background & Interaction Notes
- Summary of history, preferences, and key notes.
```

### 2. Projects (`entities/projects/`)

Active or historical projects — any bounded body of work with goals and team.

```markdown
# Project: [ProjectName]

## Metadata
- Status: [Active / Paused / Completed]
- Owner: [[lead_person]]
- Start Date: [YYYY-MM-DD]

## Objectives & Scope
- Essential Goals: [Bullet list of project goals]

## Related Entities
- Team: [[person_a]], [[person_b]]
- External Partners: [[external_entity_name]]
- Key Decisions: [[decision_name]]
- Deliverables: [[commitment_name]]
```

### 3. Decisions (`entities/decisions/`)

Architectural, strategic, or operational decisions. ADR-style: context, outcome,
consequences. Use the `Status` field to distinguish live from superseded decisions.

```markdown
# Decision: [DecisionName]

## Metadata
- Status: [Proposed / Accepted / Rejected / Superseded]
- Deciders: [[person_a]], [[person_b]]
- Date: [YYYY-MM-DD]

## Context & Problem Statement
- Why is this decision being made? What forces are at play?

## Decision Outcome
- Selected approach and why.
- Consequences, trade-offs, and downstream impacts.

## Related Links
- Project scope: [[project_name]]
- Alternatives discussed with: [[external_entity_name]]
```

### 4. Commitments (`entities/commitments/`)

Promises, deliverables, obligations, or milestones — anything someone is accountable for.

```markdown
# Commitment: [CommitmentName]

## Metadata
- Status: [Pending / Active / Fulfilled / Broken]
- Assignee: [[person_name]]
- Due Date: [YYYY-MM-DD]

## Description
- Specific deliverables, obligations, or milestones.

## Related Links
- Associated Project: [[project_name]]
```

### 5. External Entities (`entities/external/`)

External organizations, vendors, clients, API services, or strategic partners.

```markdown
# External: [ExternalEntityName]

## Metadata
- Type: [Partner / Vendor / API Service / Client]
- Primary Contact: [[person_name]]

## Engagement Scope
- Summary of agreement, API interactions, or strategic alignment.

## Related Links
- Linked Projects: [[project_name]]
- Associated Commitments: [[commitment_name]]
```

## SEMANTIC INDEX

Every entity file gets a `## SEMANTIC INDEX` section appended automatically by
DiffMem after each session. Do not edit it manually. It is a machine-readable
JSON descriptor used by the retrieval agent for fast entity triage:

```json
{"name":"alice_smith","type":"human","role":"Lead Engineer","strength":"Medium","hard_cues":["lead engineer","alice","smith"],"soft_cues":["direct communicator"],"emotional_cues":["trusted colleague"],"related_entities":["diff_mem","decide_git_storage"]}
```

## Commit Style

- Session writes: `Session {session_id} | EntityA, EntityB`
- Consolidation: `consolidate(dedupe):`, `consolidate(redistribute):`, `consolidate(link):`

## History and Temporal Reasoning

Use `git log`, `git diff`, and `git blame` to reconstruct how decisions evolved,
when commitments were made, or how a project's scope changed over time.
The timeline files provide a human-readable session-by-session narrative.
