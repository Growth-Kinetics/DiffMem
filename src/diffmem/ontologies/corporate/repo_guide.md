# DiffMem Corporate Repository Guide (v2)

This guide defines the structure, creation, and maintenance of a DiffMem memory
repository using the **corporate ontology (v2)** — a company-grounded model where
every entity is viewed through its relationship to the root company entity.
Memory files are plain-text Markdown with YAML frontmatter, in a Git repo whose
commit history captures how knowledge evolves over time.

## Architecture

Same as base DiffMem: one orphan branch per agent/user, checked out into an
isolated worktree. The difference is the folder taxonomy, the frontmatter
contract, and the company lens.

## Repository Structure

```
<worktree_root>/
├── repo_guide.md          # This file (schema reference)
├── index.md               # Autogenerated: strength-ranked entity manifest
├── followups.md           # Autogenerated: live to-do list (Open Items + Other Items)
├── {user_id}.md           # The company root entity (always present, the hub)
├── entities/
│   ├── people/            # Individuals — internal team + external contacts
│   ├── external/          # External orgs — clients, vendors, partners
│   ├── projects/          # Bounded bodies of work; OWN the Open Items section
│   └── decisions/         # ADR-style decisions
└── timeline/
    └── YYYY-MM.md         # Monthly chronological session log
```

## The Company Lens

The root entity (`{user_id}.md`) IS the company. Every other entity declares its
**vector to the company** in frontmatter, so retrieval can always traverse from
any entity back to the company and answer "how does this relate to us":

- **people** → `affiliation` (`internal` | `external_at:<client_slug>`) + `allocation`
- **external** → `relationship_to_company` (`active_client` | `prospect` | `vendor` | `partner` | `former`) + `revenue_weight`
- **projects** → `owning_engagement` (a client slug, or `internal`) + `company_priority`
- **decisions** → `deciders` + `date`

## YAML Frontmatter (structured metadata)

Every entity file begins with a YAML frontmatter block. **This is the only
location for structured/queryable fields** — there is no trailing JSON block.
The retrieval agent reads frontmatter first (progressive disclosure) and decides
whether to load the body.

```yaml
---
type: project            # REQUIRED. Matches the folder's index_type family.
title: Project Atlas
status: active            # MUST be a value from the status_enums for this type
timestamp: 2026-06-12T00:00:00Z
owning_engagement: northwind   # company-lens vector (projects)
company_priority: high
# … retrieval cue keys …
strength: Medium
hard_cues: [atlas, exec agent, attention]
soft_cues: [communication overload]
related_entities: [acme, maya_rivera]
aliases: [atlas_poc]
---
```

Required: `type`. Status fields MUST take a value from the `status_enums` for the
entity type (see `schema.json`). Code canonicalizes status before any filter —
but the writer MUST emit the canonical enum value, not freeform prose.

## Open Items (the demoted commitments)

Commitments are **no longer a top-level entity type**. A deliverable, milestone,
or follow-up lives as an entry inside its **owner** file's `## Open Items`
section — almost always the owning project, occasionally the owning person for
personal follow-ups. Each entry has a `Status` from the `open_item` enum:

```markdown
## Open Items

- **[open]** Deliver v1 API — assignee [[maya_rivera]], due 2026-07-15
  Context: scoped in the Apr 9 alignment; blocks the client demo.
- **[in_progress]** Capacity signals POC — assignee [[sam_rivera]]
- **[done]** Onboard data catalog — completed 2026-06-01
```

When an item's status reaches `done`/`cancelled` it self-evicts from the live
`followups.md` (history lives in git). Do NOT create a separate file per item.

## followups.md (live to-do list)

Two sections, regenerated on every commit:

- **Active Items** — deterministic aggregation of every `## Open Items` entry
  across all entity files whose canonical status is `open`/`in_progress`/`blocked`.
  Done/cancelled items drop automatically.
- **Other Items** — LLM-curated lightweight follow-ups that don't warrant an Open
  Items entry (e.g. "Maya to send the deck to Sam next week"). Capped ≤15; never
  duplicates an existing Open Items entry.

Safe to share with the team — it never contains historical or completed work.

## File Naming Convention

`snake_case`, all lowercase, words separated by underscores (engine enforces):
`entities/people/maya_rivera.md`, `entities/projects/atlas.md`,
`entities/decisions/decide_git_storage.md`, `entities/external/northwind.md`.

## Linking Between Entities

Obsidian-style wikilinks, one per target per file, first/prominent mention:
`[[atlas]]`, `[[maya_rivera]]`, `[[entities/external/northwind|Northwind]]`
(full path + display name when disambiguating). The frontmatter is off-limits to
link insertion.

## Entity Types and Frontmatter Templates

### people (`entities/people/`)
```yaml
---

title: Maya Rivera
status: active
affiliation: internal
allocation: 1.0
---

## Active Context & Relationships
## Background & Interaction Notes
```


## Conformance

A file in `entities/<type>/` MUST carry frontmatter `type` matching its folder's
`index_type` family (`people`→`human`, `external`→`company`, `projects`→`project`,
`decisions`→`decision`). The conformance check flags mismatches — these are the
"wrong bucket" bugs that let analysis work sit in a commitments folder.

## Commit Style

- Session writes: `Session {session_id} | EntityA, EntityB`
- Consolidation: `consolidate(dedupe):`, `consolidate(redistribute):`,
  `consolidate(link):`, `consolidate(reabsorb):`

## History and Temporal Reasoning

Use `git log`, `git diff`, `git blame` to reconstruct how decisions evolved, when
Open Items moved status, or how a project's scope changed. Timeline files provide
a human-readable session-by-session narrative.
